Purpose of This Analysis

The purpose of this page is to compare Norman’s Action Cycle and GOMS at an interaction level, using a simple and familiar task: enabling Wi-Fi on a smartphone.

This analysis follows the scope of Lab & Assignment 01 and uses selective concepts from Dix et al., Human–Computer Interaction (Chapter 9 and Chapter 12).

Norman vs GOMS Comparison

Comparison using the task: “Enable Wi-Fi on a smartphone”
Norman’s Stage User Action GOMS Mapping Interpretation
Form Goal User wants internet access Goal: Enable Wi-Fi Both models start with an explicit user goal
Form Intention Decides to use quick settings Method selection GOMS formalizes method choice using selection rules
Specify Action Searches for Wi-Fi icon Mental operator (KLM) Norman explicitly models perception and cognition
Execute Action Taps the Wi-Fi toggle Operator: Tap Direct alignment between execution and operator
Evaluate Outcome Icon changes color → Wi-Fi ON Not modeled Norman closes the Gulf of Evaluation through feedback

Evaluation Perspective (Dix et al., Ch. 9)

Norman’s model is especially useful for evaluation because it explains where interaction breakdowns occur through the Gulf of Execution and Gulf of Evaluation.

GOMS, in contrast, is not an evaluation framework; it assumes error-free expert performance and focuses on task structure rather than feedback.

Why GOMS Alone Is Not Enough

According to Dix et al. (Chapter 12), GOMS and KLM are predictive models. They estimate task efficiency but do not explain user confusion, misunderstanding, or interpretation.

Norman’s model complements GOMS by explaining why users succeed or fail during interaction.
